# UI Screen Map

## Primary navigation

- Dashboard
- Users
- Access Viewer
- Access Requests
- Access Reviews
- Reports
- Folder Templates

## 1. Dashboard

Purpose:

- show governance posture at a glance
- highlight pending approvals, expiring exceptions, and review activity
- surface recent audit-relevant events

Key sections:

- active memberships by drive
- open restricted access requests
- quarterly review campaign status
- recent audit log events
- latest generated reports

## 2. Users

Purpose:

- search Google Workspace users
- assign internal app roles
- apply or remove mapped Google Group memberships

Key sections:

- user search
- user profile summary
- assigned app roles
- mapped Google Groups preview
- membership actions
- action history

## 3. Access Viewer

Purpose:

- provide a clean effective-access view for a single user

Key sections:

- user identity summary
- current Google Groups
- inherited Shared Drive access
- restricted-folder exceptions
- effective access summary

## 4. Access Requests

Purpose:

- handle restricted-folder exception workflow only

Key sections:

- create request form
- approval queue
- approved exceptions
- rejected requests
- expiring and expired exceptions

## 5. Access Reviews

Purpose:

- run quarterly review campaigns over group-based access

Key sections:

- review campaign list
- selected campaign summary
- group/member review table
- keep/remove/needs-review actions
- export and completion controls

## 6. Reports

Purpose:

- generate, archive, and trace evidence artifacts

Key sections:

- report catalog
- generate action
- generation history
- Google Drive links
- local metadata and timestamps

## 7. Folder Templates

Purpose:

- create standard Exploration and Engineering project folders in approved locations

Key sections:

- template selector
- target parent path
- generated folder tree preview
- create action
- creation audit result

## Workflow notes

- direct per-user access screens are intentionally excluded from the main navigation
- restricted access is separated from normal assignment to reinforce the exception-only model
- audit evidence should be visible from every mutation-oriented screen
